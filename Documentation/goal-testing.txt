MEDIA TAGGING MANAGER JELLYFIN PLUGIN — TESTING AND VALIDATION TRACKER
======================================================================

Purpose
-------
This document records every test required to verify Media Tagging Manager
Jellyfin Plugin in a real Jellyfin environment before a public release. It is
separate from project-goals.txt: that file defines what the project must do;
this file records whether each behavior has actually been tested and what
happened.

When test feedback is provided, update the matching item's status and add a
short professional result to the Test result log. Keep failures, unexpected
behavior, workarounds, and retest results documented rather than deleting them.

Status key
----------

1. Installation and plugin discovery
------------------------------------

[ ] Build the Release plugin successfully from a clean checkout.
[ ] Confirm the published assembly is named Jellyfin.Plugin.MediaTaggingManager.dll.
[ ] Install the DLL manually in a dedicated Jellyfin plugin directory.
[ ] Restart Jellyfin and confirm the plugin loads without server startup errors.
[ ] Confirm the Dashboard -> Plugins entry displays "Media Tagging Manager".
[ ] Open the plugin dashboard page and confirm all three areas are available:
    Main Settings, View Tags, and Scan.
[ ] Confirm the plugin's existing stable GUID retains settings after replacing
    an older local build with a newer local build.

Acceptance criteria:
- The plugin loads and its dashboard is reachable without manually editing
  Jellyfin files after installation.
- The server remains usable and reports no plugin-load exception.

2. Library selection and settings persistence
----------------------------------------------

[~] Confirm all configured Jellyfin libraries are listed in Main Settings.
[ ] Select one test library, save, leave the page, and confirm the selection
    persists after reopening the dashboard.
[ ] Restart Jellyfin and confirm the selection persists.
[ ] Select more than one test library and confirm all selected libraries remain
    selected after saving and restart.
[ ] Confirm Save Main Settings saves the plugin's server-wide configuration
    directly and does not offer a misleading per-user or server-name scope
    prompt. Provider and network tags are shared metadata for users who can
    access the selected libraries.
[ ] Clear every selection and confirm the plugin does not interpret this as
    permission to scan every library.
[ ] Confirm an unselected library is absent from Scan controls and is never
    changed by Scan all selected libraries.
[ ] Confirm up to three TMDb availability-country selections save and reload.
[ ] Confirm provider-only mode saves and reloads.
[ ] Confirm network-only mode saves and reloads.
[ ] Confirm combined provider-and-network mode saves and reloads.
[ ] Confirm replacement of plugin-managed tags can be switched on and off.
[ ] Confirm **Here in Jellyfin** is selected by default in Tag Destination(s)
    and that at least one tag destination is required before saving.
[ ] With a selected library configured to save local NFO metadata, select
    **In my NFO files**, scan a known title, and confirm its NFO contains the
    resulting Provider:/Network: tags.
[ ] With NFO selected, confirm a scan is blocked with a clear message when any
    selected library is not configured to save local NFO metadata.
[ ] Confirm choosing both destinations writes matching tags to Jellyfin and to
    the configured NFO files, and that a backup restore returns the selected
    destination(s) to the saved tag list.

Acceptance criteria:
- Settings persist across a page reload and Jellyfin restart.
- No unselected library is queried, backed up, or modified.

3. Source configuration and credential safety
----------------------------------------------

[ ] Configure a valid TMDb API Read Access Token through the dashboard.
[ ] Configure a valid Watchmode API key through the dashboard.
[ ] Confirm a blank TMDb field disables TMDb without affecting another enabled
    source.
[ ] Confirm a blank Watchmode field disables Watchmode without affecting
    another enabled source.
[ ] Confirm the plugin refuses a manual scan when every source is disabled or
    missing credentials.
[ ] Confirm that refusal leaves existing Provider: and Network: tags unchanged.
[ ] Use an intentionally invalid source credential and confirm a failed lookup
    leaves existing plugin-managed tags unchanged.
[ ] Inspect source code, published DLL directory, backup files, and any private
    manifest used for testing; confirm no real API key is present.
[ ] Confirm source credentials are entered only in the Jellyfin dashboard and
    remain absent from README examples and release files.
[ ] Confirm normal source requests send credentials in HTTP headers rather than
    including them in request URLs, where server-side inspection is possible.
[ ] Set a Watchmode request limit and enter the account's **Quota Resets On**
    date. Confirm the visible usage counter identifies the correct 30-day cycle
    (including a reset date of 2026-08-21), does not reset at a calendar-month
    boundary, and stops title and catalog requests at the selected limit.

Acceptance criteria:
- Every server uses its own credentials.
- Missing or invalid credentials cannot erase known tags.
- No credential is exposed in project, release, backup, or manifest artifacts.

4. Provider and network tagging behavior
-----------------------------------------

[ ] Scan a title with one known provider and confirm a `Provider: <name>` tag
    is written.
[ ] Scan a title with more than one known provider and confirm all distinct
    providers are retained.
[ ] Scan a TV title with one known network and confirm a `Network: <name>` tag
    is written.
[ ] Scan a title with multiple network values where source data provides them
    and confirm all distinct networks are retained.
[ ] Confirm provider and network tags remain separate on the same title.
[ ] Confirm duplicate source values result in only one case-insensitive tag.
[ ] Confirm provider-only mode writes provider tags without adding networks.
[ ] Confirm network-only mode writes network tags without adding providers.
[ ] Confirm combined mode writes both kinds of tags where source data provides
    them.
[ ] Confirm tags unrelated to the plugin remain untouched after a healthy scan.
[ ] With replacement enabled, confirm stale plugin-owned tags are reconciled on
    a healthy source response while unrelated tags remain untouched.
[ ] With replacement disabled, confirm existing plugin-owned tags are retained
    alongside newly found tags.
[ ] Confirm a source response with no external identifier available does not
    invent a provider or network.
[ ] Confirm the configured availability countries change provider results only
    the source supports; record any expected regional difference.
[ ] Before scanning any media, confirm Select Providers loads the full TMDb
    movie/TV provider catalog for the selected regions and the Watchmode source
    catalog when the corresponding keys are configured.
[ ] Before scanning any media, confirm Select Networks loads Watchmode's full
    TV-network catalog when a Watchmode API key is configured.
[ ] After a selected-library scan, confirm newly discovered provider and
    network values remain listed even if a selection sync removes their tags.
[ ] Confirm Select All and Select None affect every provider checkbox without
    changing network checkboxes, and the equivalent network controls affect
    only networks.
[ ] Search providers and networks independently; confirm each search filters
    only its own list and does not clear existing checkbox selections.
[ ] Confirm selections remain available in those lists after a selection sync
    removes their current tags.
[ ] Select a subset of providers, run Sync with Only Selected Providers, and
    confirm a backup is created, only unselected `Provider:` tags are removed,
    no source request is made, and `Network:`/unrelated tags remain unchanged.
[ ] Select a subset of networks, run Sync with Only Selected Networks, and
    confirm a backup is created, only unselected `Network:` tags are removed,
    no source request is made, and `Provider:`/unrelated tags remain unchanged.
[ ] After each selection sync, scan again and confirm only the selected kind's
    chosen values are applied in future scans.

Acceptance criteria:
- Tags are clear, overlap-friendly, and safe to filter in Jellyfin.
- The plugin never treats a missing or failed lookup as a reason to guess or
  remove unrelated metadata.

5. Manual scans, scan status, and performance
---------------------------------------------

[ ] Click Scan all selected libraries and confirm each selected library is
    checked.
[ ] Confirm a single pre-scan backup is created before an all-selected-library
    scan changes tags.
[ ] During a multi-item scan, confirm the dashboard shows running state.
[ ] Confirm the displayed current title changes as items are processed.
[ ] Confirm completed count, total count, and percentage are credible.
[ ] Confirm the ETA appears after sufficient progress and is reasonably close
    on a multi-item test library.
[ ] Confirm the ETA never displays an invalid value such as `NaN`.
[ ] Confirm a completed scan stays visible as a completion result and reports
    checked items, newly added tags, and media items that received a new tag.
[ ] Click Stop Scan and record final status,
    partial tag state, and backup availability.
[ ] Confirm a scan error is shown as a non-fatal status and does not make the
    Jellyfin dashboard unusable.
[ ] Test a representative larger library or larger batch after small-library
    behavior is confirmed; record duration and source-rate-limit behavior.

Acceptance criteria:
- An administrator can tell what is happening during a scan.
- A failed or cancelled scan remains recoverable using backups.

6. New incoming media checks
----------------------------

[ ] Confirm incoming-media checks are off by default in a new configuration.
[ ] Enable incoming-media checks and save settings.
[ ] Run a normal Jellyfin library scan to establish the first enabled baseline.
[ ] Add a new supported movie to a selected test library.
[ ] Run Jellyfin's normal library scan and confirm the new movie is checked and
    tagged after the scan completes.
[ ] Add a new supported TV series to a selected test library.
[ ] Run Jellyfin's normal library scan and confirm the new series is checked
    and tagged after the scan completes.
[ ] Confirm an existing older title is not unnecessarily rechecked by the
    incoming-media path.
[ ] Confirm an automatic pre-change backup is created before an incoming-media
    update changes tags.
[ ] Disable incoming-media checks, add a new supported title, run Jellyfin's
    normal library scan, and confirm no automatic source lookup/tag update runs.
[ ] Confirm manual scans and scheduled full scans still work while
    incoming-media checks are disabled.
[ ] Restart Jellyfin and confirm incoming-media enabled/disabled state and its
    baseline timestamp behave correctly.

Acceptance criteria:
- Incoming media is checked only when the setting is enabled.
- Disabling the setting has no effect on manual or scheduled full refreshes.

7. Scheduled full refresh
-------------------------

[ ] Enable automatic full refresh and set a test interval.
[ ] Restart Jellyfin and confirm Refresh provider and network tags appears in
    Dashboard -> Scheduled Tasks.
[ ] Start the scheduled task manually and confirm it scans configured libraries.
[ ] Confirm the task creates a pre-change backup before changing tags.
[ ] Confirm the task does not scan unselected libraries.
[ ] Confirm disabling automatic full refresh prevents its default recurring
    trigger after restart.
[ ] Change the interval, restart Jellyfin, and confirm the task reflects the
    expected new default schedule.
[ ] Confirm task cancellation behaves safely and leaves a usable backup.
[ ] Record actual schedule behavior, duration, and source rate-limit behavior.

Acceptance criteria:
- Manual, scheduled, incoming-media, combined, and fully disabled automatic
  modes behave independently as configured.

9. Backups, restore, and undo
-----------------------------

[ ] Create a named manual backup from settings.
[ ] Confirm the backup list shows its label, timestamp, and item count.
[ ] Restart Jellyfin and confirm the backup remains listed.
[ ] Confirm the saved backup file exists in the plugin data directory.
[ ] Confirm the backup contains complete tag lists, including an unrelated test
    tag not created by this plugin.
[ ] Change provider, network, and unrelated test tags after creating a backup.
[ ] Restore the named backup and confirm all saved tags return exactly to their
    backup state.
[ ] Confirm the restore warning clearly explains that current tags are
    overwritten for saved items.
[ ] Confirm an item removed after backup is skipped without stopping the rest
    of restoration.
[ ] Run a tag-changing scan or manual edit, then use Undo last tag operation.
[ ] Confirm Undo restores the newest pre-change backup as expected.
[ ] Restore an older listed backup and confirm backup selection works, not only
    Undo latest.
[ ] Select a backup and use Delete Backup; confirm only that selected backup is
    removed and no current Jellyfin tags change.
[ ] Click Delete Backup with no backup selected; confirm the dashboard says
    "Select a backup before deleting."
[ ] Test backup and restore with a larger selected library; record duration and
    backup file size.
[ ] Confirm backups remain inaccessible to non-administrator Jellyfin users.

Acceptance criteria:
- A user can safely return selected-library tag state to a known prior backup.
- Backup files survive normal server restarts and never include API credentials.
